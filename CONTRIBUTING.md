# Contributing

## What we accept

| Contribution type | Examples |
|---|---|
| Spec improvements | Clarify requirements, add edge cases, tighten MUST/SHOULD wording |
| Test cases | Add conformance tests and expected outcomes |
| Policy schema | Extend `policy.schema.json` without breaking compatibility |
| Operational guidance | Indexer/wallet hardening patterns |

## Workflow

1. Open an issue describing the change.
2. Submit a PR referencing the issue.
3. Keep changes falsifiable; avoid vague marketing language.
